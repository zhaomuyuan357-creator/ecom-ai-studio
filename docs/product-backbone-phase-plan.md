# 产品骨架阶段计划

## 文档目的

这份文档用于记录一个新的推进阶段：

`从持续深挖工作流细节，切换到先补产品骨架闭环。`

它不是替代现有这些文档：

- `docs/detail-page-upgrade-optimization-plan.md`
- `docs/detail-page-generation-step-plan.md`

而是在它们之外，单独定义一个新的产品阶段，避免团队继续陷入：

1. 一直在前端工作流里往下钻
2. 每轮都能修一点，但整体产品闭环感不强
3. “我的 / 登录 / 数据沉淀 / 充值”长期悬空

---

## 为什么现在开这个阶段

当前项目已经证明两件事：

1. `场景图生成 + 详情页生成工作流` 是产品核心价值
2. 只继续优化工作流细节，容易不断陷入更深的局部修补

所以现在需要引入新的阶段目标：

`先把产品从“能跑的工作流页面”推进成“有数据归属、有作品沉淀、有用户入口”的最小可运营产品。`

---

## 新阶段名称

建议正式命名为：

`产品骨架闭环阶段`

也可以理解为：

`Phase A：数据库 + 我的 + 轻登录 + 额度准备`

---

## 阶段总目标

这一阶段的目标不是把商业系统一次性做满，而是先完成下面这几个最关键的骨架：

1. 用户生成过的内容可以被保存
2. 用户再次进入时，可以在“我的”里看到自己的内容
3. 用户身份和数据归属关系可以建立
4. 后续如果要接充值、套餐、额度，不需要推倒重来

---

## 核心判断

### 现在应该做的

1. 开数据库
2. 打开“我的”模块
3. 做最小登录系统
4. 为后续额度和充值预留结构

### 当前技术路线决定

这一阶段先固定为：

1. 第一版数据库使用 `SQLite`
2. ORM 使用 `Prisma`
3. 数据库操作层与业务逻辑层解耦
4. 明确保留后续切换到 `PostgreSQL` 的空间

原因：

1. 当前后端是 `Node.js + Express`
2. `Prisma` 与当前技术栈更自然
3. `SQLAlchemy` 会把系统拉到 Python 侧，增加额外复杂度
4. `SQLite` 适合第一版快速闭环
5. 只要 Repository 层和 Service 层拆开，后续切到 `PostgreSQL` 的代价可控

补充约束：

1. 当前 `Prisma` 已支持在 `SQLite` 下使用 `Enum` 和 `Json`
2. 所以第一版允许为工作流快照类数据使用 `Json`
3. 但不建议把 `String[]` 这类标量列表当成核心通用结构
4. 多表原子操作的事务必须由 `Service` 层发起，不放在 `Repository` 层分散处理

### 现在不建议一步做到头的

1. 复杂权限系统
2. 完整订单中心
3. 真正完整的支付闭环
4. 大而全的会员体系
5. 先做很多后台运营功能

---

## 本阶段推荐推进顺序

严格按下面顺序推进。

### Step A1：数据库落地

先把“生成历史”和“用户内容”存下来。

优先存这些数据：

1. 用户基本信息
2. 上传素材记录
3. 场景图生成任务
4. 详情页生成任务
5. 生成结果图片
6. 详情页工作流草稿
7. 作品集合 / 最近记录

这一步的核心不是“设计完美数据库”，而是先建立真实持久层。

### Step A2：打开“我的”模块

第一版“我的”不追求复杂，先做最小闭环。

优先包含：

1. 我的作品
2. 我的任务
3. 最近生成
4. 已上传素材

第一版目标：

`用户进入“我的”时，明确感觉这不是一次性工具，而是一个能积累作品和资产的产品。`

### Step A3：最小登录系统

登录系统先做轻量版，不做重。

推荐第一版范围：

1. 邮箱验证码登录，或
2. 手机验证码登录，或
3. 极简账号登录

这一阶段的重点不是安全体系做满，而是建立：

`用户身份 -> 数据归属 -> 我的内容`

### Step A4：额度模型预埋

先不要急着接真充值，但要提前把额度结构设计好。

至少要定义：

1. 场景图如何计次
2. 详情页首图如何计次
3. 单模块生成如何计次
4. 失败任务是否返还额度
5. 用户当前剩余额度如何表示

### Step A5：充值模块再接入

充值放在这个阶段最后。

因为它依赖前面这些前提：

1. 用户系统
2. 数据归属
3. 额度规则
4. 任务状态
5. 订单记录
6. 支付回调处理

如果这些前提没先稳定，充值会把复杂度一下拉高很多。

---

## 本阶段不替代原工作流主线

这一点必须明确：

当前已经形成的工作流主线仍然有效：

1. 摘要确认
2. 首图生成
3. 首图确认
4. 模块规划
5. 参考图归类
6. 补充真实事实
7. 单模块生成
8. 统一导出详情页

这个新阶段不是推翻它。

而是换一个更高层的推进优先级：

`先把产品骨架接起来，再继续深挖工作流细节。`

---

## 与当前阶段的关系

### 已有阶段

当前仍然存在：

- `Phase 1-5`：围绕工作流、前端结构、提示词优化、模块生成质量展开

### 新增阶段

现在新增：

- `产品骨架闭环阶段`

它和原来的关系是：

1. 原工作流阶段解决“生成是否好用”
2. 新骨架阶段解决“产品是否像一个真正产品”

---

## 本阶段的边界

### 要做

1. 数据库存储
2. 用户数据归属
3. “我的”页面第一版
4. 轻登录
5. 额度结构预留

### 不做

1. 完整支付后台
2. 复杂会员等级
3. 大规模管理后台
4. 过重的权限角色系统
5. 一开始就把充值做成正式商业系统

---

## 验收标准

这一阶段完成时，至少满足下面这些标准：

1. 用户登录后，生成记录能持久保存
2. “我的”模块可以真实查看作品和任务
3. 用户再次进入时，能看到自己的历史内容
4. 任务、作品、素材之间有明确归属关系
5. 后续要接充值时，不需要重做整个数据层

---

## 当前建议执行方式

这轮建议不要再并行做太多事情。

优先顺序建议固定为：

1. 先写数据库与产品骨架方案
2. 再定“我的”第一版结构
3. 再接最小登录
4. 再决定额度模型
5. 最后再考虑充值

---

`产品骨架实施拆解文档`
